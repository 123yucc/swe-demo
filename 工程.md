## 一、整体架构模式

### Orchestrator + Subagents（主体方向）

一个 Orchestrator agent 负责 Phase 3（Closure Check）和 Phase 4（Patch Planning），将 Phase 1/2 各步骤分发给专职 subagent。

```
Orchestrator
├── artifact-parser        (Phase 1)
├── symptom-extractor      (Phase 2, Step 1-4.1)
├── constraint-extractor   (Phase 2, Step 2-4.2)
├── localization-extractor (Phase 2, Step 5.1)
├── structural-extractor   (Phase 2, Step 5.2)
└── patch-executor         (Phase 5)
```

**优势**：每个 subagent 上下文聚焦，工具权限可精细隔离，结构直接对应理论分层。 **实现方式**：SDK 的 `agents` 参数 + `Task` 工具，或 `.claude/agents/` 目录定义。

## 二、Evidence 存储方式

### 文件系统 JSON cards

Claude Code SDK 以工作目录为基础运行，所有 agent（包括 subagent）都能通过 Read/Write/Edit 工具访问同一个文件系统。Evidence Cards 以 JSON 文件形式存储在工作目录下，成为 agent 间天然的共享状态层，不需要任何额外的通信机制。

目录结构：

```
workdir/
└── {instance_id}/                        # 每个 SWE-Bench instance 独立隔离
    ├── repo/                             # clone 下来的目标仓库
    │
    ├── artifacts/                        # Phase 1 输入
    │   ├── problem_statement.md
    │   ├── requirements.md
    │   ├── new_interfaces.md
    │   └── expected_and current_behavior.md
    │
    ├── evidence/                         # Phase 2 产出（核心）
    │   ├── symptom_card.json
    │   ├── localization_card.json
    │   ├── constraint_card.json
    │   ├── structural_card.json
    │   └── card_versions/               # 每次更新留版本，支持 diff 和 retry 回溯
    │       ├── symptom_card_v1.json
    │       ├── symptom_card_v2.json
    │       └── ...
    │
    ├── closure/                          # Phase 3 产出
    │   ├── closure_check.json           # 最新一次 closure check 结果
    │   └── closure_history.jsonl        # 每次 check 追加，便于分析多轮循环
    │
    ├── plan/                             # Phase 4 产出
    │   └── patch_plan.json
    │
    ├── patch/                            # Phase 5 产出
    │   └── {instance_id}.pred           # 最终 patch，直接兼容 SWE-Bench 格式
    │
    └── run_log.jsonl                     # Orchestrator 的决策日志
```

每个 subagent 负责写入自己的 card。Orchestrator 不直接构造 card 内容，而是给 subagent 明确的写入路径和 schema 要求。

## 四、Repair 循环的控制流  （依然待定）

### 选项 A：Evidence Gap-Guided Retry

Closure Check 返回 `gaps` 列表，Orchestrator 将 gap 转化为针对性的 retrieval 指令，重新触发对应的 subagent 补充特定 card：

```python
while not closure_passed and attempts < MAX_ATTEMPTS:
    gaps = run_closure_check(cards)
    if gaps:
        targeted_prompt = build_gap_filling_prompt(gaps)  # 只补缺失证据
        new_evidence = run_evidence_agent(targeted_prompt)
        update_cards(cards, new_evidence)
```

**优势**：直接解决你的 `stuck-in-loop` 问题——循环有明确目的（填补 gap），有明确终止条件（closure pass 或 max attempts）。

------

### 选项 B：Session Fork 探索多假设

当 attribution 存在歧义时（多个候选 root cause），用 `forkSession: true` 并行探索多个修复假设，选取 validation 通过的路径。

```python
# Fork 两个 session 分别探索不同的 localization 假设
sessions = await asyncio.gather(
    query(..., resume={"session_id": base_session, "fork": True}),
    query(..., resume={"session_id": base_session, "fork": True}),
)
```

## 五、工具权限设计

不同 Phase 的 subagent 应有不同的工具集，避免 patch agent 在 evidence 阶段就开始修改代码：

| Agent 角色          | 允许工具                           | 禁止工具          |
| ------------------- | ---------------------------------- | ----------------- |
| Artifact Parser     | Read, Glob                         | Write, Edit, Bash |
| Evidence Extractors | Read, Glob, Grep, Bash（只读命令） | Write, Edit       |
| Closure Checker     | Read（cards only）                 | 全部写操作        |
| Patch Executor      | Read, Edit, Write, Bash            | —                 |

这可以通过 `PreToolUse` hook 实现精细控制，比如拦截 `Bash` 中带写操作的命令，隔离写操作。

## 六、SWE-Bench Pro 批量评估的工程适配

### 关键设计点

**并发控制**：SWE-Bench Pro 评估脚本支持 `--num_workers`，因此我们的 harness 需要支持 instance 级别并行，每个 instance 有独立的 working directory 和 session。

**输出格式**：最终需要生成 `.pred` 文件，可在 `Stop` hook 里自动提取最终 patch 并写入：

```python
@hook("Stop")
async def export_patch(event):
    patch = extract_git_diff(event.working_dir)
    write_pred_file(instance_id, patch)
```

**断点续跑**：利用 session resume + 文件系统 cards，支持从 Closure Check 失败处重试，而不是从头重跑整个 Phase 1-2。
